- Workers complete in-flight requests within `graceful_timeout` (120s)
- The FastAPI lifespan handler runs shutdown logic (close Redis, dispose engine, etc.)
- Container exits cleanly, typically in 2-5 seconds with no active requests

<task type="auto">
  <name>Task 5: Modify SkillSandbox to accept network parameter from environment</name>
  <files>backend/app/services/skill/sandbox.py</files>
  <action>
Modify `backend/app/services/skill/sandbox.py` to accept an optional `network` parameter in `start_service_container()`. When the backend runs inside Docker Compose, skill containers must join the compose network (`nextflow-net` or equivalent) for DNS resolution. The network name comes from an environment variable `COMPOSE_NETWORK_NAME`.

Make these exact changes to `backend/app/services/skill/sandbox.py`:

1. Add `import os` at the top of the file (after the existing imports, around line 10).

2. In the `__init__` method, add a line to read the network name from environment:

   After the line `self._docker = docker.from_env()` (line 39), add:
   ```python
       self._network = os.environ.get("COMPOSE_NETWORK_NAME", "")
   ```

3. In the `start_service_container` method, replace the `network_mode` logic. Change the existing code:

   ```python
           network_mode = "bridge" if permissions.get("network") else "none"
   ```

   to:

   ```python
           network_mode = "bridge" if permissions.get("network") else "none"
           # When running inside Docker Compose, connect skill containers to
           # the compose network for DNS resolution (e.g., backend -> backend:8000)
           network_name = self._network if self._network else None
   ```

4. In the same `start_service_container` method, add the `network` parameter to the `containers.run()` call. Find the line containing `network_mode=network_mode,` and after it, add a new parameter:

   ```python
               network=network_name,
   ```

   The result should be that the `containers.run()` call now has both `network_mode=network_mode,` and `network=network_name,` parameters.

   Note: When `network` is `None` (development mode, no `COMPOSE_NETWORK_NAME` set), Docker uses the default behavior (same as before). When `network` is set (e.g., `"nextflow-net"`), the container joins that network in addition to the mode specified by `network_mode`.

Do NOT modify the `run_script` method -- script-type containers use `network_mode="none"` and don't need compose network.
  </action>

  </verify>
