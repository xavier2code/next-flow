### Task 3: Verify graceful shutdown (SIGTERM handling)

**read_first:**
- `backend/entrypoint.sh` (exec form ensures SIGTERM reaches gunicorn)
- `backend/gunicorn.conf.py` (graceful_timeout = 120)
- `backend/app/main.py` (lifespan handler has shutdown logic)

**action:**
Start the backend container again (same as Task 1), then test graceful shutdown:

```bash
# Start the container
docker run -d --name nextflow-shutdown-test \
  --network host \
  -e DATABASE_URL="postgresql+asyncpg://nextflow:nextflow@localhost:5432/nextflow" \
  -e REDIS_URL="redis://localhost:6380/0" \
  -e MINIO_ENDPOINT="localhost:9000" \
  -e MINIO_ACCESS_KEY="nextflow" \
  -e MINIO_SECRET_KEY="nextflow123" \
  -e JWT_SECRET_KEY="test-secret-key-for-verification" \
  -e CORS_ORIGINS='["http://localhost:5173"]' \
  -e OPENAI_API_KEY="" \
  nextflow-backend

# Wait for startup
sleep 15

# Verify it is running
curl -s http://localhost:8000/api/v1/health

# Send SIGTERM and time the shutdown
START=$(date +%s)
docker stop nextflow-shutdown-test
END=$(date +%s)
ELAPSED=$((END - START))
echo "Shutdown took ${ELAPSED} seconds"

# Check logs for graceful shutdown messages
docker logs nextflow-shutdown-test 2>&1 | tail -20

docker rm nextflow-shutdown-test
```

Expected behavior:
- `docker stop` sends SIGTERM to PID 1 (gunicorn via exec in entrypoint.sh)
- Gunicorn receives SIGTERM, tells workers to shut down gracefully
- Workers complete in-flight requests within `graceful_timeout` (120s)
- The FastAPI lifespan handler runs shutdown logic (close Redis, dispose engine, etc.)
- Container exits cleanly

The shutdown should complete in well under 120 seconds (typically 2-5 seconds with no active requests).

**acceptance_criteria:**
- `docker stop` completes without error
- Shutdown time is under 30 seconds (no active requests, so should be fast)
- Container logs contain `shutting_down_application` (from lifespan handler in main.py)
- Container does NOT show `SIGKILL` (which would indicate graceful shutdown failed)

### Task 5: Modify SkillSandbox to accept network parameter from environment

**read_first:**
- `backend/app/services/skill/sandbox.py` (full file -- understand start_service_container method)
- `backend/app/core/config.py` (understand Settings class and how env vars are loaded)

**action:**
Modify `backend/app/services/skill/sandbox.py` to accept an optional `network` parameter in `start_service_container()`. When the backend runs inside Docker Compose, skill containers must join the compose network (`nextflow-net` or equivalent) for DNS resolution. The network name comes from an environment variable `COMPOSE_NETWORK_NAME`.

Make these exact changes to `backend/app/services/skill/sandbox.py`:

1. Add `import os` at the top of the file (after the existing imports, around line 10).

2. In the `__init__` method, add a line to read the network name from environment:

   After the line `self._docker = docker.from_env()` (line 39), add:
   ```python
       self._network = os.environ.get("COMPOSE_NETWORK_NAME", "")
   ```

3. In the `start_service_container` method, replace the `network_mode` logic. Change the existing code:

   ```python
           network_mode = "bridge" if permissions.get("network") else "none"
   ```

   to:

   ```python
           network_mode = "bridge" if permissions.get("network") else "none"
           # When running inside Docker Compose, connect skill containers to
           # the compose network for DNS resolution (e.g., backend -> backend:8000)
           network_name = self._network if self._network else None
   ```

4. In the same `start_service_container` method, add the `network` parameter to the `containers.run()` call. Find the line containing `network_mode=network_mode,` and after it, add a new parameter:

   ```python
               network=network_name,
   ```

   The result should be that the `containers.run()` call now has both `network_mode=network_mode,` and `network=network_name,` parameters.

   Note: When `network` is `None` (development mode, no `COMPOSE_NETWORK_NAME` set), Docker uses the default behavior (same as before). When `network` is set (e.g., `"nextflow-net"`), the container joins that network in addition to the mode specified by `network_mode`.

**acceptance_criteria:**
- `sandbox.py` contains `import os` at the top
- `sandbox.py` `__init__` method contains `self._network = os.environ.get("COMPOSE_NETWORK_NAME", "")`
- `sandbox.py` `start_service_container` method contains `network=network_name,` in the `containers.run()` call
- `sandbox.py` contains the comment `# When running inside Docker Compose, connect skill containers to`
- Existing `network_mode=network_mode` line is NOT removed (both parameters coexist)
- Existing `run_script` method is NOT modified (script-type containers use `network_mode="none"` and don't need compose network)
